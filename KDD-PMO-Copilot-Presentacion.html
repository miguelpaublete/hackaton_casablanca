⚠️ El problema

El knowledge se pierde
en cada reunión

En proyectos tecnológicos complejos, el 80% del conocimiento clave se genera en reuniones y nunca se estructura. El PMO tiene que convertir horas de reunión en documentación accionable — a mano, sin herramientas, sin tiempo.

🕐

Tiempo perdido en documentación manual

El PMO dedica horas a transcribir, estructurar y formatear decisiones y tareas que ya se discutieron en reunión.

📉

Knowledge que se evapora

Las decisiones clave (ADRs), el contexto de negocio y los riesgos identificados nunca llegan a un repositorio consultable.

🔀

Falta de trazabilidad

¿Por qué se tomó esa decisión hace 3 semanas? ¿Quién era el owner de esa tarea? Imposible saberlo sin specs estructuradas.

🤖

IA sin contexto de negocio

GitHub Copilot y otros LLMs no conocen el dominio del proyecto. Sin specs estructuradas, la IA trabaja a ciegas.

¿Qué es Knowledge-Driven Development (KDD)?

KDD es una metodología que convierte el conocimiento de negocio en specs estructuradas y versionadas, creando una fuente única de verdad que alimenta tanto a los desarrolladores como a los agentes de IA. Cada reunión produce artifacts persistentes (ADRs, Domain specs, Work Tasks) que se acumulan como contexto RAG nativo para GitHub Copilot y otros LLMs.

🏗️ Arquitectura

Cómo está construido

Arquitectura modular en Python con una interfaz web, un motor de IA y un sistema de versionado en Git.

📁
Capa de Input — /actas/
PDFs y TXTs de reuniones organizados por proyecto (Deco algo, Carbon Markets, Pagos Internacionales). El watcher detecta nuevos archivos automáticamente.
↓
🔄
Capa de Conversión — acta_loader.py
Convierte PDFs a TXT limpio (pdfminer/pdfplumber), normaliza el texto y lo prepara para el prompt de la IA. Evita procesar actas ya convertidas.
↓
🧠
Capa de IA — extractor.py + GitHub Models (GPT-4.1)
Envía el texto al LLM con un prompt KDD estructurado. El modelo genera ADRs, Domain Specs y Work Tasks en formato Markdown con frontmatter YAML válido.
↓
🖥️
Capa de Validación — app.py (Streamlit)
Interfaz web donde el PMO selecciona proyecto, elige acta, lanza la extracción y valida/edita cada artefacto antes de commitear. Puerto 8501.
↓
📦
Capa de Persistencia — committer.py + /specs/
Escribe los Markdown en /specs/[proyecto]/[tipo]/, hace git add + commit + push al repo compartido con proxy corporativo BBVA y token OAuth.
↓
📧
Capa de Notificación — watcher.py + notifier.py (Mailjet)
Proceso background que detecta actas nuevas y envía email HTML al PMO con resumen de archivos detectados y enlace directo a la interfaz.
📋 Tipos de artefactos KDD generados

Taxonomía de tres ejes

Knowledge (persistente) · Governance (decisiones) · Work (efímero por sprint)

ADR
Architecture Decision Record
Decisiones de arquitectura con contexto, razonamiento y consecuencias
DOM
Domain Spec
Conocimiento de dominio de negocio estructurado y persistente
WRK-TASK
Work Task
Tarea accionable con owner, criterios de aceptación y plan de test
WRK-SPEC
Work Spec
Especificación técnica para una funcionalidad o cambio concreto
ARCH
Architecture Spec
Decisiones de diseño técnico de alto nivel
WRK-PLAN
Work Plan
Plan de trabajo con hitos, dependencias y estimaciones
📅 Cómo lo construimos

De la idea al producto
en una hackaton

El proceso de desarrollo que seguimos durante la Hackaton Casablanca.

FASE 1 — Definición del problema
Identificamos el pain point del PMO
Reunión de análisis donde identificamos que el proceso de documentación de reuniones consumía horas de trabajo manual y el knowledge acababa en emails sin estructura ni trazabilidad.
FASE 2 — Diseño del framework KDD
Creamos la taxonomía de artefactos
Definimos el framework Knowledge-Driven Development: tres ejes ortogonales (Knowledge, Work, Governance), frontmatter YAML estándar, y las reglas de validación de specs.
FASE 3 — Prototipo del agente
Extractor + Streamlit básico
Construimos el primer extractor con Vertex AI (Gemini) y la interfaz Streamlit. Primeras pruebas con actas reales del proyecto Deco Algo. Problemas con la lectura de PDFs → añadimos conversión a TXT.
FASE 4 — Integración con GitHub
Commit automático al repo compartido
Implementamos committer.py con soporte para proxy corporativo BBVA (407 Proxy Auth). Migramos de Vertex AI a GitHub Models (GPT-4.1) para simplicidad de autenticación. Resolvimos problemas de git config (user.email).
FASE 5 — Notificaciones automáticas
Watcher + Email al PMO via Mailjet
Añadimos el watcher que vigila /actas/ y el notifier que envía emails HTML al PMO con Mailjet (API HTTP, funciona detrás del proxy corporativo). Configuramos sender verificado en alejandro.montalban@nfq.es.
FASE 6 — Producción y validación real
28 specs generadas y commiteadas
Procesamos 4 actas reales del proyecto Deco Algo generando ADR-001 a ADR-006, DOM-RISK-001 y WRK-TASK-001 a 028. Todos los artefactos validados por PMO y presentes en el repositorio compartido.
🛠️ Manual de instalación

Ponlo en marcha en
tu proyecto

Guía paso a paso para que cualquier PMO pueda instalar y usar KDD PMO Copilot en menos de 30 minutos.

📋

Prerequisitos

Python 3.11+ instalado · Git configurado · Cuenta GitHub con acceso al repositorio · Token GitHub con permisos Contents: Read & Write + Models: Read · Cuenta Mailjet (gratuita) con email verificado

1

Clonar el repositorio

Clona el repositorio del proyecto en tu máquina local.

# Clonar con proxy corporativo (red BBVA)
git -c "http.proxy=http://USUARIO:PASSWORD@proxyvip.igrupobbva:8080" \
    clone https://github.com/miguelpaublete/hackaton_casablanca.git

cd hackaton_casablanca/agent
2

Instalar dependencias Python

Instala todas las librerías necesarias desde el requirements.txt.

# Recomendado: crear entorno virtual
python -m venv venv
.\venv\Scripts\activate        # Windows
source venv/bin/activate         # Mac/Linux

# Instalar dependencias
pip install -r requirements.txt
3

Configurar el archivo .env

Copia el ejemplo y rellena tus credenciales. Nunca subas el .env a GitHub.

# Copiar plantilla
cp .env.example .env

Edita el .env con estos valores:

Variable Descripción Req.
GITHUB_TOKEN Fine-grained PAT con permisos Models: Read ✦ REQ
GITHUB_COMMIT_TOKEN Token con Contents: Read & Write en el repo destino ✦ REQ
GITHUB_REPO usuario/nombre-repo (ej: miguelpaublete/hackaton_casablanca) ✦ REQ
MAILJET_API_KEY API Key de tu cuenta Mailjet ✦ REQ
MAILJET_API_SECRET Secret Key de tu cuenta Mailjet ✦ REQ
SENDER_EMAIL Email verificado en Mailjet como remitente ✦ REQ
PMO_EMAIL Email del PMO que recibirá las notificaciones ✦ REQ
HTTP_PROXY http://USUARIO:PASS@proxyvip.igrupobbva:8080 ◎ OPT
STREAMLIT_PORT Puerto de la interfaz web (por defecto: 8501) ◎ OPT
4

Configurar git con tu identidad

Git necesita saber quién eres para poder hacer commits. Configúralo en el repositorio local.

git config user.email "tu.email@bbva.com"
git config user.name "Tu Nombre"
5

Preparar la carpeta de actas

Organiza tus actas de reuniones en la carpeta /actas/ con una subcarpeta por proyecto. El nombre de la carpeta será el nombre del proyecto en la interfaz.

actas/
├── Deco algo/
│   ├── reunion-2026-03-09.pdf
│   └── reunion-2026-03-23.txt
├── Carbon Markets/
│   └── kickoff-2026-03-31.pdf
└── Mi Proyecto/
    └── weekly-2026-05-01.txt

💡 Los PDFs se convierten automáticamente a TXT antes de procesarse. Los TXTs se usan directamente.

6

Lanzar la interfaz

Arranca la interfaz web Streamlit. Se abrirá automáticamente en tu navegador.

# Desde la carpeta agent/
streamlit run app.py

# O con puerto específico
streamlit run app.py --server.port 8502

Accede en: http://localhost:8501

7

Activar el watcher de notificaciones (opcional)

Lanza el proceso en background que monitoriza /actas/ y envía emails automáticos al PMO.

# En una terminal separada
python watcher.py

# Verificar que el email funciona
python send_test_email.py
8

¡Listo! Genera tu primera spec

En la interfaz: 1) Selecciona proyecto → 2) Elige acta → 3) "Extraer artefactos" → 4) Revisa → 5) "Validar y Commitear" ✅

# Flujo completo en la UI:
Proyecto    → seleccionar carpeta de actas
Acta        → elegir PDF o TXT de la lista
Extraer     → la IA genera los artefactos (~30s)
Validar     → revisar y editar si hace falta
Commitear  → push automático a GitHub ✅
⚠️

Configuración de proxy en red BBVA

Si estás en la red corporativa de BBVA, necesitas configurar el proxy para que los commits lleguen a GitHub. El sistema lo hace automáticamente si tienes HTTP_PROXY configurado en el .env con el formato http://T0XXXXX:PASSWORD@proxyvip.igrupobbva:8080. La variable de tu usuario Windows es tu matrícula BBVA (ej: T057294).
