01
Getting Started
Your first spec in 10 minutes
What is KDD?

Knowledge-Driven Development (KDD) is a methodology where structured specifications are the primary artifact that drives software delivery. Instead of ad-hoc documents or tribal knowledge, every significant decision, domain rule, and architectural constraint lives as a versioned, machine-readable spec.

Three Axes

Knowledge (persistent)

ARCH, DOM, PROD, FEAT, DOC — what the organization knows. Persists across projects.

Work (ephemeral)

WRK-SPEC, WRK-PLAN, WRK-TASK — what a team does. Scoped to a delivery.

Governance (bridge)

RFC, ADR, RULE — how knowledge evolves through decisions.

Your First Spec in 10 Minutes

Step 1: Create the file

mkdir -p specs/domain
touch specs/domain/DOM-PAYMENTS-001-transfer-limits.md

Step 2: Write the frontmatter

---
id: DOM-PAYMENTS-001
type: knowledge
layer: domain
title: Domestic Transfer Limits
status: draft
confidence: medium
version: "0.1.0"
owner: payments-team
domain: payments
subdomain: transfers
tags: [limits, compliance]
dependencies: []
---

Step 3: Write the body

## Intent

Define the maximum transfer amounts for domestic payments,
ensuring compliance with AML thresholds.

## Definition

### Transfer Limits

| Customer Tier | Single Transfer | Daily Aggregate |
|---------------|---------------:|----------------:|
| Standard      |        €10,000 |         €25,000 |
| Premium       |        €50,000 |        €100,000 |
| Corporate     |       €500,000 |      €1,000,000 |

### Rules

- Transfers exceeding the single limit require dual approval.
- Daily aggregate is calculated on a rolling 24-hour window.
- Corporate limits may be overridden per-client via RULE artifacts.

## Acceptance Criteria

- [ ] Payment service enforces single-transfer limits by tier
- [ ] Daily aggregate check runs before authorization
- [ ] Dual approval flow triggers above threshold

## Evidence

- AML regulation reference: EU Directive 2015/849
- Business validation: Product owner sign-off pending

Step 4: Validate

node apps/spec-graph/spec-graph.mjs --specs specs validate

Progressive Adoption

LevelNameWhat you doWhat you get
L1DocumentWrite specs in Markdown with YAML frontmatterSingle source of truth, version control
L2ValidateRun spec-graph validate, enforce in CIBroken links caught early, dependency integrity
L3AutomateGenerate boilerplate from specsConsistency between spec and code
L4GenerateAI agents consume specs as contextDomain-aligned generation, less hallucination
L5OrchestrateAgents activate knowledge automatically per taskFull agentic workflow with knowledge loop

Start at L1. Most teams get value within the first sprint. See Section 09: Adoption Levels for detailed criteria and examples.

02
Create a Knowledge Spec
ARCH, DOM, PROD, FEAT, DOC — step by step

Knowledge Artifacts capture persistent organizational knowledge — architecture decisions, domain rules, product vision, feature definitions, and documentation. They outlive any single project.

Step 1: Choose the Type and Layer

LayerType IDUse when...Example
ArchitectureARCHDefining cross-cutting technical decisionsEvent-driven architecture, API gateway pattern
DomainDOMCapturing business rules and domain logicVaR calculation rules, transfer limits
ProductPRODDescribing product-level capabilitiesPortfolio analytics platform vision
FeatureFEATSpecifying a concrete featureReal-time P&L dashboard
DocumentationDOCRecording conventions, guides, standardsAPI naming conventions, onboarding guide
Constrains many features?

→ ARCH

Domain expert owns it?

→ DOM

Product owner owns it?

→ PROD

User-facing behavior?

→ FEAT

Explains how to do something?

→ DOC

Step 2: Create the Frontmatter

---
id: DOM-RISK-001              # TYPE-AREA-NNN
type: knowledge               # always "knowledge" for this axis
layer: domain                 # architecture | domain | product | feature | documentation
title: Market Risk Calculation (VaR)
status: draft                 # draft → active → deprecated
confidence: medium            # low | medium | high
version: "0.1.0"
owner: risk-analytics-team
domain: markets               # functional domain
subdomain: risk-management    # subdomain within the vertical
tags: [var, market-risk, regulatory]
dependencies:
  - id: ARCH-002
    type: constrained-by      # implements | constrained-by | extends | uses-data-from
  - id: DOM-REG-001
    type: implements
reviewers: []
---

Required Fields

FieldDescription
idUnique identifier: TYPE-AREA-NNN
typeAlways knowledge
layerOne of: architecture, domain, product, feature, documentation
titleHuman-readable title
statusdraft → active → deprecated
confidencelow (inferred), medium (validated OR reviewed), high (validated AND reviewed)
versionSemver string
ownerTeam or person responsible

Step 3: Write the Body

Every Knowledge spec follows this structure:

Intent

Why does this spec exist? 2–3 sentences. The most important section.

Definition

The actual content. Structure depends on the layer: rules, decisions, capabilities, behavior, or procedures.

Acceptance Criteria

How do we know it's correctly implemented? Write as a checklist.

Evidence

What validates this spec? Test results, regulatory references, sign-offs.

Step 4: Declare Dependencies

RelationMeaningExample
implementsThis spec implements a higher-level specFEAT implements PROD
constrained-byMust respect constraints from anotherDOM constrained-by ARCH
extendsAdds detail to another at the same layerDOM-002 extends DOM-001
uses-data-fromConsumes data defined in anotherFEAT uses-data-from DOM
depends-onGeneric dependencyAny layer
supersedesReplaces another specNew version of deprecated spec

Step 5: Validate

node apps/spec-graph/spec-graph.mjs --specs <your-specs-dir> validate

Common issues: missing required frontmatter fields, dependencies referencing non-existent spec IDs, invalid status or confidence values.

Step 6: PR Review

  • Run through the spec checklist before submitting
  • Explain why this spec exists (not just what it contains)
  • Tag the owner and relevant reviewers
  • If this spec changes dependencies, note the impact (use spec-graph impact <id>)

After merge, update confidence: Reviewed + validated → high. Reviewed OR validated → medium. Neither → low.

03
Create Work Artifacts
WRK-SPEC → WRK-PLAN → WRK-TASK

Work Artifacts capture what a team is doing right now. They are ephemeral — scoped to a delivery and archived when done.

WRK-SPEC (what & why)  →  WRK-PLAN (how)  →  WRK-TASK (do)

The Three Levels

ArtifactPurposeOwnerLifecycle
WRK-SPECDefines scope, constraints, and activated knowledgeTech lead / architectDraft → Active → Completed → Archived
WRK-PLANDecomposes into tasks, defines architecture approachTech leadDraft → Active → Completed → Archived
WRK-TASKAtomic unit of work, implementable by one person/agentDeveloperDraft → Active → Completed → Archived

Step 1: Create a WRK-SPEC

The WRK-SPEC answers: What are we building and why? What knowledge do we need?

---
id: WRK-SPEC-001
type: work
layer: spec
title: VaR Engine Redesign
status: draft
version: "0.1.0"
owner: risk-analytics-team
activates:
  - DOM-RISK-001      # Domain rules we must respect
  - ARCH-002          # Architecture constraints
  - DOM-REG-001       # Regulatory requirements
tags: [var, performance, redesign]
---

The activates field is the most important: it declares which Knowledge Artifacts provide context for this work.

WRK-SPEC Body Structure

Problem Statement

Why are we doing this? What's broken or missing?

Proposed Solution

High-level approach (not implementation details).

Activated Knowledge

For each activated spec, explain why it's relevant.

Constraints & Acceptance

Non-negotiable boundaries + how we know work is done.

Step 2: Create a WRK-PLAN

The WRK-PLAN answers: How will we build it? What are the tasks?

---
id: WRK-PLAN-001
type: work
layer: plan
title: VaR Engine Implementation Plan
status: draft
version: "0.1.0"
owner: risk-analytics-team
parent: WRK-SPEC-001           # Links to the parent WRK-SPEC
activates:
  - DOM-RISK-001
  - ARCH-002
tags: [var, implementation]
---

Key sections: Architecture Approach, Task Decomposition (table of WRK-TASKs with estimates), Constraints, Risk Assessment, Dependencies.

Step 3: Create WRK-TASKs

Each WRK-TASK is an atomic unit of work — implementable by one developer or AI agent.

---
id: WRK-TASK-001
type: work
layer: task
title: VaR Calculation Service
status: draft
version: "0.1.0"
owner: developer-a
parent: WRK-PLAN-001
scope:
  includes:
    - src/var/calculation/**
  excludes:
    - src/var/api/**
tags: [var, core-engine]
---

Key sections: Objective, Technical Specification, Acceptance Criteria, Test Strategy.

Quick Reference

QuestionAnswer
How many WRK-TASKs per WRK-PLAN?3–8 is typical. More suggests the WRK-SPEC scope is too large.
Can I skip WRK-PLAN?For very small work (1–2 tasks), you can go WRK-SPEC → WRK-TASK directly.
Who writes each level?WRK-SPEC: architect/lead. WRK-PLAN: lead. WRK-TASK: developer or lead.
Can AI agents write WRK-TASKs?Yes — this is a key L4/L5 pattern. The agent uses activated knowledge as context.
04
Governance Cycle
RFC → SPEC → ADR

The governance cycle is how knowledge evolves. It ensures that changes to organizational knowledge are proposed, discussed, decided, and recorded — not applied ad hoc.

RFC (proposal)  →  Discussion  →  Decision  →  SPEC (new/updated) + ADR (record)

When to Use What

SituationArtifact
Proposing a new approach or significant changeRFC
The decision is straightforward and uncontestedGo directly to SPEC
Recording why a decision was madeADR
Defining a standing constraint or policyRULE

Rule of thumb: If the change affects multiple teams or reverses a previous decision, write an RFC first. If it's a local improvement within your domain, update the SPEC directly.

Writing an RFC

An RFC (Request for Comments) proposes a change and invites discussion.

---
id: RFC-001
type: governance
layer: rfc
title: Migrate from REST to gRPC for Internal Services
status: proposed           # proposed → accepted → superseded | rejected
version: "0.1.0"
owner: platform-team
tags: [api, grpc, migration]
dependencies:
  - id: ARCH-001
    type: supersedes
---

RFC Body Sections

  • Context — What is the current situation?
  • Proposal — What specifically are you proposing?
  • Rationale — Why this approach over alternatives?
  • Alternatives Considered — What else was evaluated?
  • Impact — Which specs change? Which teams are affected?
  • Open Questions — What needs discussion?

RFC Lifecycle

  1. Proposed: Author creates and shares with stakeholders
  2. Discussion: Team reviews, comments, suggests modifications
  3. Decision: Accept, reject, or defer
  4. Accepted → Creates/updates SPECs + records an ADR
  5. Rejected → Record the ADR explaining why (knowledge is preserved)

Writing an ADR

An ADR captures why a decision was made. Written after the decision.

---
id: ADR-001
type: governance
layer: adr
title: Use Event-Driven Architecture for Trade Lifecycle
status: accepted           # accepted → superseded
version: "1.0.0"
owner: platform-team
dependencies:
  - id: ARCH-002
    type: depends-on
---

ADR Body Sections

  • Status — Accepted / Superseded + date
  • Context — Situation requiring a decision
  • Decision — One clear sentence
  • Rationale — Factors weighted
  • Consequences — Positive, negative, risks
  • Alternatives Considered — Table with rejection reasons

Writing a RULE

A RULE is a standing constraint — more durable than ADRs, defines ongoing policy.

---
id: RULE-001
type: governance
layer: rule
title: All Services Must Publish Domain Events
status: active
version: "1.0.0"
owner: architecture-guild
---

Sections: Rule (the constraint), Rationale, Enforcement (how it's checked), Exceptions (documented as ADRs).

Keep RFCs focused

One proposal per RFC. Don't bundle unrelated changes.

Record rejected proposals

Write ADRs even for rejected proposals — the reasoning prevents re-litigating.

Link everything

RFCs reference affected specs. ADRs reference produced specs. Use dependencies.

05
Contextual Activation
Inject the right knowledge into work

Contextual activation is the process of injecting the right organizational knowledge into a work stream. It ensures that when a team (or AI agent) works on a task, they have the relevant domain rules, architecture constraints, and product context.

Why It Matters

Without activation

Developers re-discover constraints by trial and error. AI agents hallucinate domain rules. Architecture decisions are violated.

With activation

Right knowledge available before implementation. Constraints respected by design. AI agents produce domain-aligned output.

The Activation Pipeline

1. Explicit  →  2. Transitive  →  3. Filtered  →  4. Budgeted

1. Explicit Activation

Start with the specs that directly define what you're building. Ask: "Which specs describe the domain rules, architecture patterns, or product requirements I need?"

2. Transitive Activation

Follow the dependency graph from your explicitly activated specs:

node apps/spec-graph/spec-graph.mjs --specs <dir> context DOM-RISK-001 --depth 2

3. Filtered Activation

Remove specs that describe a different aspect, are at the wrong abstraction layer, or are deprecated. Filter heuristic: If removing a spec wouldn't change how you implement the work, it doesn't belong.

4. Budgeted Activation

Work levelTypical budget
WRK-SPEC5–10 specs
WRK-PLAN3–7 specs (subset)
WRK-TASK2–5 specs (only what this task touches)

Activation Matrix

PhasePrimary layersSecondary layers
RequirementsPROD, DOMARCH
DesignARCH, DOMPROD, FEAT
BuildFEAT, DOMARCH
TestDOM, FEATARCH
DeployARCHDOM

CLI Commands for Activation

# Filter specs by criteria
node apps/spec-graph/spec-graph.mjs --specs <dir> filter --layer domain --tag risk

# Full context around a spec (depth 2 = direct + transitive)
node apps/spec-graph/spec-graph.mjs --specs <dir> context DOM-RISK-001 --depth 2

# Impact analysis: what depends on a spec you're changing?
node apps/spec-graph/spec-graph.mjs --specs <dir> impact ARCH-002

# Shortest path between two specs
node apps/spec-graph/spec-graph.mjs --specs <dir> path DOM-RISK-001 FEAT-ANALYTICS-001

Example: VaR Engine Redesign

StepActionResult
Explicit"We're redesigning the VaR engine"DOM-RISK-001, ARCH-002
Transitivecontext DOM-RISK-001 --depth 2+ DOM-REG-001, ARCH-001, DOM-DATA-001
FilteredARCH-001 too abstract, DOM-DATA-001 different dataRemove both
Budgeted3 specs is within budgetFinal: DOM-RISK-001, ARCH-002, DOM-REG-001
06
Consolidation
Closing the Work → Knowledge loop

Consolidation is the process by which completed work feeds back into organizational knowledge. Without it, teams learn things during implementation that never make it back into specs.

Knowledge ──activates──→ Work ──consolidates──→ Knowledge (updated)

The Consolidation Checklist

Run through this when closing a WRK-SPEC:

1. Architecture Decisions → ADRs

"Did we make any significant technical decisions during this work?"

Common triggers: chose between approaches, rejected original plan, discovered an uncaptured constraint, made a trade-off.

2. Domain Learnings → Updated Knowledge Specs

"Did we discover domain rules, edge cases, or invariants not in existing DOM specs?"

  • Minor additions: Update existing DOM spec (bump version)
  • New subdomain: Create a new DOM spec
  • Correction: Update spec, note correction in Evidence

3. Architecture Patterns → Updated ARCH Specs

"Did we establish patterns that should be reused?"

4. Product/Feature Insights → Updated PROD/FEAT Specs

"Did we learn something about user needs or product behavior?"

5. Open Questions → Resolved or Escalated

Answered → Capture in the appropriate Knowledge Artifact. Still open → Create RFC or new WRK-SPEC.

Example: VaR Engine Consolidation

DiscoveryActionArtifact
Partition by asset class outperformed portfolioWrite ADRADR-003
Cache invalidation requires event-driven TTLUpdate ARCH-002Add pattern
Regulatory VaR needs specific rounding rulesUpdate DOM-RISK-001Bump to v1.2.0
0.01% precision divergence is acceptableWrite RULERULE-005
Intraday recalculation still openCreate new workWRK-SPEC-002
Don't skip this step

Block 1–2 hours for consolidation at the end of each WRK-SPEC.

Tech lead owns it

The lead has the best view of what was learned, even if developers contribute.

Small and frequent

If a WRK-TASK reveals something important, update knowledge immediately.

08
Spec Checklist
Pre-PR validation

Frontmatter

  • id is present and follows the pattern (TYPE-AREA-NNN or WRK-TYPE-NNN)
  • type is one of: knowledge, work, governance
  • layer matches the type
  • title is present and descriptive
  • status is a valid lifecycle value
  • confidence is set (knowledge artifacts only)
  • version follows semver
  • owner is set to a real team or person

Dependencies

  • Every dependencies[].id references an existing spec
  • Every dependencies[].type uses a valid relation
  • No circular dependencies
  • Relations make semantic sense (e.g., FEAT doesn't constrain ARCH)

Work Artifacts (additional)

  • activates lists at least one Knowledge Artifact (WRK-SPEC)
  • parent references an existing Work Artifact (WRK-PLAN, WRK-TASK)
  • scope defined with includes/excludes (WRK-TASK)

Body — Knowledge Artifacts

  • Intent section explains why the spec exists
  • Definition section contains the actual specification
  • Acceptance Criteria has testable items
  • Evidence references validation sources

Body — Work Artifacts

  • Problem Statement or Objective is clear
  • Activated Knowledge explains why each spec is relevant (WRK-SPEC)
  • Task Decomposition lists all tasks with estimates (WRK-PLAN)
  • Acceptance Criteria are specific and testable

Body — Governance Artifacts

  • Context explains the situation
  • Decision/Proposal is clearly stated
  • Rationale explains reasoning
  • Alternatives Considered are documented

Validation

  • CLI validation passes: spec-graph validate
  • No orphaned specs: spec-graph orphans

Review Readiness

  • PR description explains why this spec exists
  • owner and reviewers are tagged
  • If modifying existing spec: version bumped
  • If superseding: old spec marked deprecated

Adoption Levels
L1–L5 with criteria and examples

KDD is designed for incremental adoption. Each level builds on the previous one.

L1
Document
L2
Validate
L3
Automate
L4
Generate
L5
Orchestrate

L1 — Document

What you do: Write specs as Markdown with YAML frontmatter. Store in version control.

What you get: Single source of truth, version history, searchable knowledge base.

Criteria

  • Team has a specs/ directory
  • Critical domain rules captured as DOM specs
  • Architecture decisions recorded as ARCH or ADR
  • Specs follow standard anatomy
  • Specs reviewed in PRs

Timeline: 1–2 sprints to establish.

L2 — Validate

What you do: Run automated validation on specs. Enforce in CI.

What you get: Broken dependencies caught before merge, frontmatter consistency enforced.

Criteria (in addition to L1)

  • spec-graph validate runs in CI and blocks PRs
  • spec-graph orphans runs as warning
  • Dependency graph is visualizable
  • Team reviews graph periodically

Timeline: 1 sprint after L1.

L3 — Automate

What you do: Generate code artifacts from specs — API stubs, schemas, test scaffolds.

What you get: Guaranteed consistency between spec and implementation.

Criteria (in addition to L2)

  • At least one code generation pipeline exists
  • Generated artifacts not manually edited
  • Templates for common spec types exist

Timeline: 2–3 sprints.

L4 — Generate

What you do: AI agents consume specs as context to produce code, tests, documentation.

What you get: Domain-aligned AI output, faster delivery, consistent architecture.

Criteria (in addition to L3)

  • WRK-SPEC → WRK-PLAN → WRK-TASK flow is standard
  • activates field used consistently
  • AI agents receive activated specs as context
  • Agent output reviewed against acceptance criteria

Timeline: 2–4 sprints after L3.

L5 — Orchestrate

What you do: Agents autonomously navigate the knowledge graph, plan work, execute tasks, and consolidate learnings.

What you get: End-to-end knowledge-driven delivery with continuous evolution.

Criteria (in addition to L4)

  • Agents run full activation pipeline
  • Consolidation partially automated
  • Governance cycle integrated into agent workflow
  • Human review gates at key checkpoints

Timeline: Long-term vision. 2–3 quarters after L4.

10
Brownfield Adoption
Adopt KDD in existing projects
The Challenge

Existing projects have domain knowledge scattered across wikis, Slack threads, and people's heads. Architecture decisions were never recorded. Code embodies rules nobody documented.

Strategy: Start Where It Hurts

Don't try to document everything. Start with the knowledge that causes the most pain when missing.

Phase 1: Critical Domain Rules (Weeks 1–2)

Identify the 3–5 domain rules that cause bugs when developers get them wrong, require asking a specific person every time, or are different from what you'd naively assume. Write them as DOM specs.

Phase 2: Architecture Decisions (Weeks 3–4)

Mine existing decisions from git history, meeting notes, senior team members. Write retroactive ADRs.

Phase 3: Validation (Week 5)

Set up spec-graph validate in CI. You're now at L2.

Phase 4: Integrate with Workflow (Ongoing)

New features get WRK-SPECs. Architecture changes go through RFC → ADR. Bug fixes that reveal undocumented rules produce new DOM specs.

Coexistence with Existing Docs

Existing formatKDD approach
Confluence/wikiExtract rules into DOM specs. Keep wiki for narratives.
Swagger/OpenAPIAlready specs — add ARCH/FEAT frontmatter that references them.
Code commentsExtract business rules into DOM specs. Leave impl comments.
README filesKeep as entry points. Add links to specs.
Architecture diagramsReference from ARCH specs. Spec = why, diagram = visual.

Anti-Patterns to Avoid

Big bang spec-writing

Don't stop feature work to document everything. Adopt incrementally.

Specifying the obvious

If a rule is clear from the code, it doesn't need a spec. Spec the surprising and complex.

Gold-plating specs

A 10-line DOM spec capturing a critical rule is more valuable than a 500-line one nobody reads.

Measuring Success

After 1 month

  • At least 5 critical domain rules captured
  • New team members reference specs instead of Slack
  • At least one bug caught because a spec existed
  • Architecture decisions from last quarter have ADRs

After 3 months

  • CI validates spec integrity
  • WRK-SPECs reference activated knowledge
  • The consolidation habit is forming

Vertical Taxonomy
Build a taxonomy for your industry

A vertical taxonomy is a hierarchical organization of domains and subdomains specific to an industry. It provides naming structure for spec IDs and browsing structure for knowledge discovery.

Step-by-Step

Step 1: Identify Functional Domains

Start with 3–6 top-level business areas. Ask stakeholders: "What are the major areas of our business?"

Insurance
├── Underwriting
├── Claims
├── Policy Administration
├── Distribution
├── Actuarial
└── Regulatory & Compliance

Step 2: Decompose into Subdomains

3–8 subdomains per domain. Ask domain experts: "Within [domain], what are the distinct capability areas?"

Claims
├── First Notice of Loss (FNOL)
├── Claims Assessment
├── Fraud Detection
├── Settlement
└── Subrogation

Step 3: Add Cross-Cutting Capabilities

Cross-Cutting
├── Client Lifecycle (KYC, onboarding)
├── Regulatory & Compliance
├── Data Management
└── Operations

Step 4: Define the Technical Domain Tree

Shared across verticals — describes technical patterns, not business domains:

Technical
├── Architecture Patterns (application, integration, data, security)
├── NFR Catalog (performance, reliability, security, operability)
├── Technology Standards (languages, platforms, data stores)
└── Quality Standards (code, API, testing, documentation)

Step 5: Create the Activation Matrix

Map which knowledge layers activate at each phase of work.

Spec as Prompt
Use specs as context for AI agents
The Problem

When AI agents generate code without domain context, they hallucinate business rules, violate architecture constraints, and produce generic patterns. Each regeneration may produce different results.

The Pattern

Feed activated specs into the agent's context window as structured input. The spec becomes the system-level context that constrains the agent's output.

Activated Knowledge (DOM, ARCH) + WRK-TASK → Agent → Domain-aligned code

Implementation Patterns

Pattern 1: Activation Bundle as System Prompt

Concatenate activated specs into a single context block. When: Simple workflows, single-agent tasks, L4 adoption.

# Activated Knowledge

## DOM-RISK-001: Market Risk Calculation (VaR)
[full spec body]

## ARCH-002: Event-Driven Architecture
[full spec body]

---

# Your Task

## WRK-TASK-001: VaR Calculation Service
[full task body]

Implement this task respecting the activated knowledge above.

Pattern 2: Layered Context Injection

Structure context by priority — domain rules first, then architecture, then task. When: Complex tasks where priority ordering matters.

# Domain Rules (MUST follow)
[DOM specs — business rules, calculations, invariants]

# Architecture Constraints (MUST respect)
[ARCH specs — patterns, technology decisions, NFRs]

# Product Context (SHOULD align with)
[PROD/FEAT specs — user outcomes, success metrics]

# Task
[WRK-TASK with acceptance criteria]

Pattern 3: Spec Reference with Retrieval

Inject summaries, let agent request full specs on demand. When: L5 adoption, large knowledge graphs, tool-use agents.

# Available Knowledge (summaries)
- DOM-RISK-001: VaR uses 250-day historical simulation
- ARCH-002: Services communicate via Kafka events
- DOM-REG-001: MiFID II reporting, T+1 deadline

# Full specs available via tool: read_spec(id)

# Task
[WRK-TASK body]

Context Window Optimization

Spec sectionInclude?Why
FrontmatterYes (summary)ID, type, confidence — helps agent understand role
IntentYesEssential for understanding purpose
DefinitionYesCore content the agent needs
Acceptance CriteriaYesDefines the contract
EvidenceOptionalOnly if agent needs validation approach
TraceabilityNoNot useful for generation

Budget guideline: A typical WRK-TASK activation (2–5 specs) should fit in 3,000–8,000 tokens.

Validation Loop

Agent generates → Run acceptance criteria → Pass? Done : Feed failure back → Iterate
Frontmatter matters

Include confidence: high vs low — signals how much to trust the spec.

Don't over-activate

5 focused specs > 20 loosely related ones. More context ≠ better output.

Fix activation, not prompts

If agents get rules wrong, activate a missing spec — don't rewrite the prompt.
